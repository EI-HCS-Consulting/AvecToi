**⚠️ Points d'attention OTA/EAS Update, à ne pas reproduire :**
- **Quoting bash (résolu, PR #86) :** ne jamais interpoler `${{ github.event.head_commit.message }}` directement dans un `run:` bash — un message multi-lignes avec guillemets casse le quoting et `eas update` échoue silencieusement (run rouge dans Actions, mais rien ne bloque le merge ni n'alerte dans l'app). Toujours passer par une variable d'environnement (`env: COMMIT_MESSAGE: ...` puis `"$COMMIT_MESSAGE"` entre guillemets dans le script). Réflexe après un merge important : `gh run list --workflow=eas-update-preview.yml --limit 3` pour confirmer un run vert.
- **Redémarrage complet nécessaire après publication :** `expo-updates` (config par défaut) télécharge la mise à jour en arrière-plan mais sert encore l'ancien bundle JS au lancement en cours — il faut fermer l'app depuis le multitâche (pas juste la mettre en arrière-plan) et parfois relancer deux fois juste après une publication.
- **Nouveau (21/07) — la liste des Updates dans le dev-launcher peut afficher une mise à jour plus récente sans que l'app l'ait réellement chargée.** Ouvrir l'écran "Updates" du dev-launcher ne suffit pas : il faut explicitement **taper/sélectionner** l'entrée la plus récente pour qu'elle se charge (le chargement peut prendre longtemps). Tant que ce n'est pas fait, l'app continue de tourner sur l'ancien bundle même si le nouveau apparaît dans la liste — source de fausses alertes "bug de code" alors que c'est juste un bundle périmé côté device de test.
- **Technique de diagnostic découverte le 21/07 — comparer les manifests EAS Update directement :** en cas de doute sur le bundle réellement chargé par un device, interroger l'URL `updates.url` du projet (`app.json`, ex. `https://u.expo.dev/305abfe3-5d04-449f-9b06-8975f2196588`) via curl avec les headers du protocole Expo Update (`Accept: multipart/mixed`, `expo-platform: android`, `expo-protocol-version: 1`, `expo-runtime-version: <valeur de app.json>`, `expo-channel-name: preview`) pour obtenir le vrai dernier manifest du channel (avec son `expo-update-id` et sa date de création), et le comparer à l'update actuellement connecté sur le device (visible dans l'écran Updates du dev-launcher). Permet de prouver objectivement un bundle périmé plutôt que de deviner.

## 2. État actuel

**Bug 1 — bundle OTA périmé sur le device de test (résolu, procédural) :** le device de l'utilisateur listait bien une mise à jour plus récente dans l'écran Updates du dev-launcher, mais n'avait pas réellement chargé cette mise à jour (elle était seulement listée). Prouvé objectivement en comparant, via curl, le manifest actuellement connecté sur le device avec le vrai dernier manifest du channel `preview` (technique détaillée dans "État global" ci-dessus). Résolu par l'utilisateur en sélectionnant explicitement la nouvelle entrée dans le dev-launcher (chargement long).

**Bug 2 — "Mes Patients" ne se rafraîchissait pas après un pivot/switch (PR #88, mergée) :** une fois sur le bon bundle, l'utilisateur a rejoint 2 nouveaux espaces via "Rejoindre un nouveau patient" mais "Mes Patients" restait vide. Root cause : le `useEffect` qui charge `linkedSpaces` depuis Supabase ne dépendait que de `[role, telephone]`, qui ne changent jamais lors d'un pivot/switch — la liste n'était donc jamais rechargée. Fix : ajout de `space?.id` aux dépendances.

**Refonte UI + Bug 3 — clic sur un patient sans effet (PR #89, mergée) :** demande explicite de l'utilisateur — déplacer "Mes Patients" dans l'accordéon (sous "Mes informations", même mécanique que le reste), afficher de vrais boutons avec le nom de chaque patient (badge "Actif" sur l'espace courant), et intégrer le bouton "➕ Rejoindre un nouveau patient" dans ce même sous-menu. À cette occasion, l'utilisateur a signalé que cliquer sur le nom d'un patient ou sur la flèche ne changeait plus d'espace. Root cause : `switchingSpaceId` n'était jamais réinitialisé après `saveVisitorSession` + `router.replace` — combiné au fait que le navigateur `Tabs` d'Expo Router ne démonte pas les écrans déjà visités, l'état restait bloqué au retour sur l'onglet Compte et désactivait le bouton. Fix : `try/finally` autour de `handleSwitchLinkedSpace` pour garantir la réinitialisation de `switchingSpaceId`.

**Dernière action avant ce handoff :** mise à jour de `Documentation/Documentation Fonctionnalités.docx` (§7.1 et §7.3) pour refléter le nouveau sous-menu déroulant avec boutons nominatifs et badge "Actif", et rédaction de ce handoff.

## 4. Ce qui a échoué
- Rien de bloquant côté code. Piste de diagnostic initiale erronée : avant d'obtenir le détail précis du device (update ID connecté, channel, date de publication), l'hypothèse de départ était un mismatch de channel/build (development vs preview) — écartée une fois les informations exactes fournies par l'utilisateur, remplacée par la vérification directe des manifests EAS Update.
- Rappel technique déjà noté dans un handoff précédent, reconfirmé : `curl` vers un endpoint EAS Update doit envoyer `Accept: multipart/mixed` (pas `application/json`), sous peine de 406 ; et le bon endpoint pour interroger le "dernier manifest d'un channel" est l'URL `updates.url` de `app.json` avec le header `expo-channel-name`, pas `https://u.expo.dev/manifest`.
