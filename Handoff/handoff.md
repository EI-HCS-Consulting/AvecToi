**⚠️ Points d'attention OTA/EAS Update, à ne pas reproduire :**
- **Quoting bash (résolu, PR #86) :** ne jamais interpoler `${{ github.event.head_commit.message }}` directement dans un `run:` bash — passer par une variable d'environnement (`env: COMMIT_MESSAGE: ...`).
- **Redémarrage complet nécessaire après publication :** fermer l'app depuis le multitâche (pas juste arrière-plan), parfois relancer deux fois juste après une publication OTA.
- **Changement d'asset natif = pas livrable en OTA.** Icône, splash, etc. exigent un nouveau EAS Build.
- **⚠️ Nouveau (22/07, résolu PR #97) — EAS Update ne remet JAMAIS le menu dev-client (shake) en place, même si le build natif l'a.** Le menu shake (`DevLauncherActivity`) est compilé dans le binaire natif une fois pour toutes au moment du `eas build` ; un `eas update` ne republie que le bundle JS et ne peut ni l'ajouter ni le retirer. Piège identifié cette session : `eas.json` combinait `developmentClient: true` avec un override `"gradleCommand": ":app:assembleRelease"` sur le profil `preview` — cette combinaison neutralise silencieusement l'inclusion du module dev-launcher, sans erreur ni avertissement au build. Fix : supprimer l'override `gradleCommand` (PR #97). Vérifié par inspection binaire de l'APK (extraction `AndroidManifest.xml` + `classes*.dex`, recherche de la chaîne `DevLauncherActivity`), pas seulement par lecture de la config — c'est la méthode fiable en cas de doute sur un futur build.

Cause : une vue Postgres s'exécute par défaut avec les droits de son propriétaire, donc les policies RLS des tables sous-jacentes sont évaluées comme le créateur de la vue et non comme l'appelant. Pas de fuite active actuellement (RLS déjà permissive sur `visitor_profiles`/`patient_spaces`, par design de ce projet — accès contrôlé côté client via PIN/token), mais risque latent si la RLS venait à être resserrée un jour. Fix : `create or replace view ... with (security_invoker = true) as ...` (Postgres 15+). Migration déployée manuellement par l'utilisateur via le Dashboard Supabase, alerte confirmée disparue. Mergée (merge commit `2a1033e`).

## 4. Ce qui a échoué
- Hypothèse initiale de l'utilisateur ("l'ai-je encore cassé dans la réécriture du code") écartée après vérification par `git diff` — aucune trace du bug dans le lot PR #96 lui-même ; la vraie cause était antérieure (PR #95) et dormante.
- Première inspection binaire du build `5f3090c1` (post-fix) a donné un faux négatif ("DevLauncherActivity absent") — en réalité un artefact de manipulation de fichiers : le dossier d'extraction `/tmp/apk-check` contenait encore un `AndroidManifest.xml` périmé de l'inspection précédente (build défectueux), lu par erreur au lieu du nouveau fichier extrait dans un sous-dossier. Corrigé en vidant/réextrayant proprement avant de relire. **Leçon : toujours purger les artefacts d'extraction précédents avant de comparer un "avant" et un "après" dans le même dossier scratch.**
- Un premier essai d'écriture du script Python d'édition du `.docx` via un heredoc Bash (`cat > fichier << 'PYEOF' ... PYEOF`) a échoué avec une erreur de parsing shell (`unexpected EOF while looking for matching`) avant même l'exécution du script — cause probablement liée au volume de texte français riche en apostrophes/guillemets typographiques dans le corps du heredoc. Contourné en écrivant le script via l'outil `Write` (paramètre structuré, pas de parsing shell) plutôt que via un heredoc Bash.
- Dans ce même script, une première tentative d'insertion de nouvelles lignes de tableau utilisait `row._tbl` (attribut inexistant sur un objet `_Row` de python-docx) au lieu de `table._tbl` — `AttributeError` corrigée en récupérant le `<w:tbl>` depuis l'objet `Table` retourné par la fonction de recherche, pas depuis la ligne.
